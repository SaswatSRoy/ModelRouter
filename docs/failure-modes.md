# Failure Modes and Degraded Behavior

See [ARCHITECTURE.md](../ARCHITECTURE.md) for system context and module breakdown, [routing.md](routing.md) for the routing pipeline, and [security.md](security.md) for privacy and auth constraints referenced below.

## Design Principle

ModelRouter follows a **"fail-open where safe, fail-closed where required"** philosophy.

The **hot path** (request serving via `router-ingress` → `router-core` → `provider-adapters`) must never fail due to a management-plane (`router-admin`, PostgreSQL) or analytics-plane (Kafka, OTel Collector, Langfuse) outage. These subsystems degrade gracefully — stale data, delayed analytics, dropped telemetry — but the gateway continues routing requests with the best information available.

**Privacy and auth constraints are fail-closed** — they are structural invariants, not scoring signals, and are **never relaxed** under degraded conditions. A `LOCAL_ONLY` request will not be routed to a cloud provider even if every local provider is down; it will fail with `ALL_CANDIDATES_EXHAUSTED` rather than violate the privacy tier. An unauthenticated request will be rejected regardless of infrastructure state.

## Failure Mode Table

| Component Failure | Impact | Degraded Behavior | Recovery |
|---|---|---|---|
| **Single provider outage** (5xx, timeout, connection refused) | Requests targeting that provider fail on first attempt. | Circuit breaker opens for the affected `(provider, model)`. `ExecutionEngine` falls back to the next candidate in the ranked list. **Transparent to client** if a fallback candidate succeeds — response is indistinguishable from a first-attempt success. Latency increases by the failed attempt's timeout + fallback invocation time. | Circuit breaker enters `HALF_OPEN` after cooldown period, admits trickle traffic. On sustained success, transitions to `CLOSED`. Background health prober accelerates recovery detection. See [routing.md](routing.md) §Circuit Breaking. |
| **All providers for a request fail** | Request cannot be served. | `ExecutionEngine` exhausts the `ExecutionPlan` and returns `ALL_CANDIDATES_EXHAUSTED` to the client with **per-candidate failure detail** (provider, model, error code, latency). Client receives a clear, actionable error — not a generic 500. | Individual providers recover independently via their circuit breakers. No global recovery gate. |
| **Redis unavailable** | Live provider scores, prompt cache, and shared rate-limit counters are inaccessible. | **Scoring:** L1 in-process cache serves stale provider scores (seconds-old, bounded by L1 TTL). Routing continues with slightly outdated latency/error-rate data — acceptable for short outages. **Prompt cache:** Bypassed entirely (every request is a cache miss). No errors surfaced to client; cost and latency increase. **Rate limiting:** Falls back to pod-local approximate counters. Per-tenant limits are enforced per-pod rather than globally — limits are less precise but never absent. **Circuit breaker state:** Falls back to pod-local state; blast radius of a single provider failure may temporarily widen across pods (each pod tracks its own error rate independently). | On Redis reconnection, shared state resynchronizes automatically. L1 caches re-warm on next read. No manual intervention required. |
| **PostgreSQL unavailable** | Policy store and admin API are inaccessible. | **Routing:** Continues with **last-known policies**, cached at gateway boot and refreshed on every Kafka-driven hot-reload. Existing policies remain effective indefinitely. **Admin API:** Returns `503 Service Unavailable`. Operators cannot create, update, or delete policies, providers, or tenants until recovery. **Audit log:** Admin mutations are blocked (not silently dropped), so the audit trail has no gaps. | On PostgreSQL recovery, admin API resumes. No policy re-sync needed — gateways already hold the latest cached version and will pick up new changes via the normal hot-reload path. |
| **Kafka unavailable** | Usage events, cache-invalidation signals, and score-update events cannot be published. | **Usage events:** Buffered in-process in a **bounded ring buffer**. On overflow, **oldest events are dropped** (newest-wins). Analytics and cost metering are delayed but request serving is completely unaffected — Kafka is off the hot path by design ([ARCHITECTURE.md](../ARCHITECTURE.md) §6). **Score updates:** Provider scores are updated from real-time request telemetry on the local pod but not propagated cross-pod until Kafka recovers. Slight inter-pod score divergence is tolerable. **Cache invalidation:** Policy changes applied via admin API will not propagate to gateways until Kafka recovers — gateways serve the last-known policy (same as PostgreSQL-unavailable behavior for the hot path). | On Kafka recovery, buffered events are flushed. Events dropped due to overflow are permanently lost — downstream consumers (analytics, billing) must tolerate gaps. No manual intervention needed for the routing path. |
| **OTel Collector unavailable** | Traces and metrics cannot be exported. | Traces and metrics are **dropped silently** (fire-and-forget export). The OTel SDK's export path is asynchronous and non-blocking — a collector outage causes zero latency impact and zero errors on the request path. Prometheus scrape endpoint on the gateway pod remains available (metrics are still computed locally via Micrometer). | On collector recovery, new traces/metrics resume export. The gap period has no retroactive backfill — it is a permanent observability gap. |
| **Langfuse unavailable** | LLM-specific trace export (prompt content, token usage, cost) fails. | Export **silently fails**. No impact on request serving. No impact on OTel operational traces — Langfuse export is a parallel, independent path ([observability.md](observability.md) §LLM-Native Tracing). | On Langfuse recovery, new traces resume. Gap period data is not retroactively exported. |
| **Gateway pod crash** | In-flight requests on that pod fail. Buffered-but-unflushed usage events on that pod are lost. | **Stateless by design** — remaining gateway pods continue serving all traffic. Kubernetes restarts the crashed pod automatically. **No request state is lost** globally; only in-flight requests on the crashed pod fail. Clients retry (client-side retry is expected; SDKs implement it). The restarted pod re-caches policies from PostgreSQL and scores from Redis on boot — no warm-up coordination needed. | Kubernetes restart + readiness probe pass. Pod rejoins the Service load-balancing pool. Typical recovery: seconds. |
| **Admin pod crash** | Policy changes, provider registration, and tenant management are unavailable. | **Hot path is completely unaffected.** Gateways use cached policies and do not depend on the admin pod at runtime. Operators cannot make configuration changes until the admin pod recovers. | Kubernetes restart. Admin pod reconnects to PostgreSQL and resumes serving the admin API. No state reconciliation needed — PostgreSQL is the source of truth. |
| **Network partition between gateway and providers** | Affected providers are unreachable from partitioned gateway pods. | Circuit breakers open for all affected `(provider, model)` pairs on the partitioned pods. **Remaining reachable providers serve traffic.** If the partition is asymmetric (some pods can reach the provider, others cannot), pod-local circuit breaker state diverges — partitioned pods stop routing to the provider while non-partitioned pods continue. This is correct behavior: each pod routes based on its own reachability. | When the partition heals, circuit breakers on affected pods enter `HALF_OPEN` and recover via the normal probe-and-promote path. |
| **Slow provider** (high latency but not failing) | Requests routed to the slow provider succeed but take longer. User-perceived latency increases. | `ProviderScorer` latency EWMA increases, causing the scoring engine to **de-prioritize** the slow provider in future routing decisions. The effect is gradual and proportional — a slightly slow provider drops in rank; a severely slow provider is effectively deprioritized to last-resort. **No hard cutoff** unless latency exceeds the policy's `maxLatencyMs`, at which point the candidate is excluded by `CandidateEnumerator`. | When the provider's latency recovers, the EWMA decays back toward normal, and the provider is re-prioritized. No manual intervention. |
| **Provider rate-limit hit (429)** | Request to that provider is rejected by the provider. | Classified as a **retryable failure**. `ExecutionEngine` falls back to the next candidate in the ranked list. The `capacityHeadroom` score component is updated to reflect the exhausted provider, **preventing future routing** to that provider until headroom recovers. Transparent to client if fallback succeeds. | Provider's rate-limit window resets (typically seconds to minutes). `capacityHeadroom` score recovers via passive telemetry from subsequent successful requests or active health probes. |
| **Cache poisoning / stale cache** | Clients could receive outdated or incorrect cached responses. | **Structurally mitigated.** Cache entries are TTL-bounded — maximum staleness is bounded by the configured TTL. Cache keys incorporate the resolved `RoutingPolicy` constraints (`privacyTier`, `requiredCapabilities`, normalized request content hash), so **cross-policy pollution is structurally impossible** ([routing.md](routing.md) §Caching Interaction). A `LOCAL_ONLY` response is never served from a cache entry generated under `CLOUD_ALLOWED`, even if the prompt content is identical. | TTL expiry automatically evicts stale entries. For immediate invalidation, Kafka-driven cache-invalidation events are published on policy changes. |
| **Streaming mid-response failure** | Provider fails after the first response chunk has been sent to the client. | **Transparent failover is impossible in v1** once the first chunk is sent — the client has already begun consuming a response from a specific provider and model. The client receives a **truncated stream terminated with an error event** (not a silent disconnect). The error event includes the provider, model, and failure reason. **This is a documented v1 limitation.** | Client-side handling required: SDKs surface the error event for application-level retry. Phase 2+ investigation may add client-side replay semantics for streaming fallback ([routing.md](routing.md) §Retry & Fallback). |

## Invariants Under Degradation

The following guarantees hold **unconditionally**, regardless of which components are degraded or unavailable:

1. **Privacy tier enforcement (`LOCAL_ONLY`) is never relaxed.** `LOCAL_ONLY` is a structural exclusion applied at `CandidateEnumerator` — it removes cloud providers from the candidate list before scoring or strategy evaluation. It is not a scoring signal that can be outweighed. A `LOCAL_ONLY` request will fail with `ALL_CANDIDATES_EXHAUSTED` rather than be routed to a cloud provider. See [security.md](security.md) §Data Handling & Privacy.

2. **Authentication and authorization are never bypassed.** A request with invalid or missing credentials is rejected at `router-security` regardless of the state of Redis, PostgreSQL, Kafka, or any provider. Auth evaluation depends only on the request itself and the security filter chain configuration (loaded at boot), not on external infrastructure availability.

3. **Per-tenant isolation is maintained.** One tenant's failure (exhausted budget, rate-limited, all preferred providers down) never affects another tenant's routing. Cache keys, rate-limit counters, score aggregates, and cost budgets are all **namespaced per tenant**. There is no shared state path through which one tenant's traffic pattern can influence another tenant's routing outcome.

4. **Circuit breaker state degrades gracefully, never catastrophically.** Circuit breaker state is shared via Redis for cross-pod consistency. If Redis is unavailable, each pod falls back to **pod-local circuit breaker state**. This may temporarily widen the blast radius (a failing provider might receive a few more requests across pods before each pod independently opens its breaker), but it **cannot cause cascading failure** — each pod's breaker still protects it independently.

## Cascading Failure Prevention

ModelRouter's architecture includes several deliberate design decisions to prevent a single-component failure from cascading into a system-wide outage:

- **Per-(provider, model) circuit breakers, not global.** A single degraded model on one provider does not take healthy models on the same provider — or any other provider — out of rotation. Blast radius is contained to the smallest possible unit. See [routing.md](routing.md) §Circuit Breaking.

- **Kafka off the hot path.** Usage metering, analytics, and cross-pod score propagation flow through Kafka, but the synchronous request-serving path never blocks on a Kafka produce call. A Kafka outage degrades analytics and cross-pod consistency, never request serving. See [ARCHITECTURE.md](../ARCHITECTURE.md) §6.

- **Stateless gateway pods.** All routing-relevant state (policies, scores, health) is externalized to Redis and PostgreSQL, cached locally for performance. No pod holds unique state that would be lost on crash or that other pods depend on. Horizontal scaling and rolling deploys are safe by construction.

- **Bounded in-process buffers for event queuing.** When Kafka is unavailable, usage events are buffered in a fixed-size ring buffer. The buffer does not grow unboundedly — on overflow, the oldest events are dropped. This prevents an infrastructure outage from causing OOM on gateway pods.

- **Rate limiting at ingress before the routing engine.** Per-tenant rate limits are enforced at `router-ingress`, before a request reaches `PolicyResolver`, scoring, or any provider adapter. A misbehaving or compromised tenant is rejected early, protecting both ModelRouter's routing engine and downstream provider rate limits and cost exposure. See [security.md](security.md) §Rate Limiting & Abuse Prevention.

- **Backpressure via WebFlux reactive streams.** The entire hot path is built on Project Reactor (Spring WebFlux). Reactive streams propagate backpressure from providers through the gateway to clients — if a provider slows down, the gateway does not buffer unboundedly or spawn threads; it applies backpressure to the client connection. This prevents slow providers from exhausting gateway resources.
