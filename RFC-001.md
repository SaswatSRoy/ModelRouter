## 1. Summary

This RFC proposes the founding design of ModelRouter: an AI Inference Control Plane that decouples applications from inference providers via policy-driven routing. It documents *why* the project exists, the design philosophy behind it, the alternatives we considered and rejected, the tradeoffs we are consciously accepting, our assumptions, and how the design is expected to evolve. It also answers the critical design questions that must be resolved before Phase 1 implementation begins.

## 2. Why ModelRouter Exists

Teams integrating multiple LLM providers converge on the same ad-hoc solution: provider-selection `if/else` chains embedded in application code, duplicated per service, untested until a provider outage exposes the gap. This is the same problem HTTP microservices had before API gateways and service meshes existed — and the same category of solution applies: extract the cross-cutting concern (routing, retries, observability, policy) into shared infrastructure that applications call through, not around.

Nothing in the current OSS landscape treats this as a **control plane** problem. Most existing tools are either (a) a thin reverse proxy that normalizes request/response shapes across providers but does no policy-driven decision-making, or (b) an SDK-level abstraction that still runs *inside* the application process, so it doesn't solve duplication, centralized observability, or hot-reloadable policy. ModelRouter is deliberately infrastructure: a separately deployed, independently scaled service that owns the routing decision.

## 3. Design Philosophy

1. **Applications express intent, not destination.** A request says "complete this, under $X, within Yms, keep it on-prem if sensitive" — never "call OpenAI's gpt-4o." The provider is an implementation detail resolved at request time.
2. **The domain core knows nothing about any specific provider.** Ports & Adapters is non-negotiable; it's what makes "add a provider in an afternoon" true rather than aspirational.
3. **Reliability is structural, not incidental.** Retries, fallback, and circuit breaking are first-class concerns of the routing engine, not something bolted on per-adapter.
4. **Observability is a design input, not an afterthought.** Every routing decision must be explainable after the fact: which candidates were considered, which was chosen, why, and what it cost.
5. **Boring where it counts.** We choose proven, operationally well-understood technology (Spring, Kafka, Redis, Postgres, Kubernetes) over novel infrastructure. The routing *policy* is where we innovate; the plumbing should not be a source of surprises.
6. **Extensibility over completeness.** v1 will not support every provider or every routing strategy. It must, however, make adding either trivial and safe.

## 4. Alternatives Considered

### 4.1 Language: Go vs. Java

**Considered:** Go, given its dominance in cloud-native infrastructure (Kubernetes, Envoy, Istio are all Go) and its lightweight concurrency model.

**Decision: Java 21.**

**Rationale:**
- Java 21's virtual threads (Project Loom) close most of the concurrency-ergonomics gap with Go, while keeping a mature, batteries-included ecosystem (Spring, extensive provider SDKs, enterprise auth/observability integrations) that a small early-stage OSS project benefits from disproportionately.
- Much of the target adopter base (per the tech stack — Spring Security, PostgreSQL, Kafka, enterprise Kubernetes deployments) already operates JVM infrastructure; a JVM-native control plane integrates into existing service meshes, secret managers, and CI/CD with less friction than a Go binary would for that specific audience.
- The DDD/Hexagonal architectural style this project commits to is more idiomatically supported by Java's type system and Spring's dependency injection than by Go's more minimalist approach — this is a stylistic tradeoff, not an objective one.

**Tradeoff accepted:** JVM startup time and memory footprint are worse than Go's, which matters more for edge/sidecar deployment than for a centrally-deployed control plane. If ModelRouter later needs a sidecar/edge form factor (e.g., an in-cluster local-inference router with sub-second cold start), that may justify a Go-based "ModelRouter Lite" as a separate project rather than a rewrite.

### 4.2 Web Framework: WebFlux vs. Spring MVC

**Decision: WebFlux.**

**Rationale:** Inference calls are long-lived, I/O-bound, and frequently streamed token-by-token. Under Spring MVC's thread-per-request model, 1000+ concurrent in-flight requests (many streaming, holding a thread for the request's full duration) would require a proportionally enormous thread pool, with poor backpressure characteristics. WebFlux's non-blocking I/O lets a small number of event-loop threads service many concurrent long-lived connections, which is exactly the traffic shape ModelRouter serves.

**Tradeoff accepted:** Reactive code is harder to write, debug, and onboard new contributors to than imperative MVC code. We accept this cost because it is concentrated in `router-ingress` and `router-core.execution` — the two modules where it's load-bearing — not spread across the whole codebase. Provider adapters and admin/CQRS-side modules, which are not on the hot streaming path, may use simpler blocking patterns internally where it doesn't hurt overall throughput (e.g., wrapped via `Mono.fromCallable` + bounded elastic scheduler), trading a small amount of purity for contributor accessibility.

*(Note: Java 21 virtual threads narrow this gap further. We are choosing WebFlux now for backpressure and streaming ergonomics, but will re-evaluate whether virtual-threads-on-MVC becomes sufficient once the ecosystem around structured concurrency matures — see §7, Future Evolution.)*

### 4.3 Routing Execution Model: Synchronous vs. Asynchronous Decision-Making

**Considered:** Making the routing decision itself asynchronous/event-driven (e.g., publish a request event, have a separate routing service consume it and dispatch).

**Decision: Synchronous, in-process routing decision; asynchronous side effects only.**

**Rationale:** The routing decision (policy resolution → scoring → strategy → candidate selection) must complete within the <30ms overhead budget and is fundamentally a request/response computation — introducing a message broker into the decision path would add latency and failure modes for no benefit. What *is* asynchronous: usage metering, cache population, and score updates derived from the outcome, none of which the client needs to wait for. This is the CQRS boundary described in ARCHITECTURE.md §1.

### 4.4 Provider Loading: Static vs. Dynamic

**Considered:** Java's `ServiceLoader`/OSGi-style dynamic plugin loading, allowing providers to be added/removed at runtime without a redeploy.

**Decision: Static compilation for v1 (each adapter is a Gradle module, wired via Spring's component scanning at boot), with adapter *configuration* (credentials, endpoints, enabled/disabled, routing weight) dynamic via `router-admin` + hot-reload.

**Rationale:** True dynamic class loading (hot-swapping adapter *code* at runtime) introduces classloader isolation, versioning, and security complexity disproportionate to the actual need — new providers are added by developers who can ship a new gateway image; what genuinely needs to change at runtime is *configuration* (is this provider enabled, what's its API key, what's its routing weight), which we already support via the CQRS admin path. We revisit true dynamic loading only if a concrete use case (e.g., a plugin marketplace) emerges — see Stretch Goals in ROADMAP.md.

### 4.5 Caching: Redis-only vs. Multi-tier

**Decision:** Redis as the shared L2 cache (prompt/response and provider-score cache), with a short-TTL (seconds) in-process L1 cache in each gateway pod to absorb the most latency-sensitive, highest-frequency reads (live provider scores) without a network hop on every single request.

**Tradeoff accepted:** L1/L2 introduces eventual consistency — a pod may act on a provider score up to a few seconds stale. This is acceptable because provider scores are inherently a moving average of recent behavior, not a value requiring strong consistency.

### 4.6 API Surface: REST vs. gRPC-only

**Decision:** REST (OpenAI-compatible-ish shape where sensible, for adoption ease) as the primary v1 surface, with gRPC as a fast-follow for internal/SDK-to-gateway traffic where streaming performance matters most (notably the Android SDK, where connection overhead and binary framing matter on mobile networks).

## 5. Tradeoffs Accepted (Summary)

| Tradeoff | We accept it because |
|---|---|
| JVM footprint > Go footprint | Ecosystem fit and target-adopter alignment outweigh raw footprint for a centrally deployed control plane |
| Reactive complexity in hot-path modules | Backpressure and streaming correctness at 1000+ concurrent requests matters more than contributor ramp-up speed for those specific modules |
| Eventual consistency on provider scores | Scores are a moving average, not a source of truth; staleness on the order of seconds is invisible to routing quality |
| Static provider compilation (v1) | Avoids classloader/security complexity; new providers still ship fast via normal release cadence |
| No hosted multi-tenant SaaS in v1 | Focus effort on making the self-hosted core excellent before building a business model around it |

## 6. Assumptions

- Adopters run Kubernetes (or are willing to) — the deployment model is not optimized for bare-VM or serverless-first environments in v1.
- Adopters have (or are willing to run) Redis, PostgreSQL, and Kafka — no attempt is made to support a "zero external dependencies" mode in v1; an embedded/lite mode is a plausible future variant, not a v1 goal.
- The majority of early traffic is text completion/chat-style inference, not multi-modal; multi-modal routing (image/audio-aware provider capability negotiation) is designed for extensibility but not optimized in v1.
- Consumers of the API are backend services and mobile apps (hence Java + Android SDKs first), not primarily browser-based frontends calling ModelRouter directly.

## 7. Critical Design Questions

### Why WebFlux over MVC?
Answered in §4.2. Streaming + high concurrency + backpressure requirements make non-blocking I/O the correct default for the hot path; blocking code remains acceptable in non-hot-path modules.

### Why Java instead of Go?
Answered in §4.1. Ecosystem fit, virtual threads closing the concurrency gap, and target-adopter infrastructure alignment.

### Should routing be synchronous?
Yes, for the decision itself (§4.3). Side effects (metering, cache population, score updates) are asynchronous via Kafka.

### How should streaming work?
`router-ingress` exposes SSE (REST) and server-streaming gRPC. Internally, `InferenceProvider.invokeStreaming()` returns a `Flux<InferenceChunk>` that flows through cache (write-through, populated once the stream completes), retry/fallback logic (which can only apply *before* the first chunk is emitted — once bytes have reached the client, a fallback would require client-side replay semantics, which is out of scope for v1 and explicitly documented as a limitation), and normalization, before being re-framed to the client's chosen protocol.

### Should routing strategies be pluggable?
Yes — `RoutingStrategy` is a port in `router-core.strategy` (§ARCHITECTURE.md 3.2). Shipping strategies (lowest-latency, cheapest-viable, weighted round robin, privacy-constrained) are first-party implementations of the same interface a third party could implement.

### Should providers be loaded dynamically?
No, not at the code level, for v1 — see §4.4. Configuration is dynamic; code is statically compiled per release.

### Should provider metadata be cached?
Yes. Static metadata (context window, supported modalities, pricing table) is cached with a longer TTL (minutes) and refreshed via `router-admin`; live metadata (current latency, error rate, health) is cached with a short TTL (seconds) as described in §4.5.

### How should retries work?
Retry policy is part of `RoutingPolicy`, resolved per request (max attempts, backoff strategy, which failure classes are retryable — e.g., 429/503/timeout are retryable, 400/401 are not). Retries are scoped to the *pre-first-byte* window; see streaming answer above for why. Retries consume the ranked candidate list produced by the strategy, advancing to the next candidate on exhaustion of a given provider's retry budget — this is what makes retry and fallback the same mechanism rather than two.

### How should health checks work?
Two tiers: **passive** (every real request's outcome updates the provider's live score/circuit-breaker state — the primary signal) and **active** (a low-frequency background prober, per provider, to detect recovery from an open circuit before real traffic would). Health state lives in Redis, shared across gateway pods.

### How should provider scoring work?
A composite, weighted score per (provider, model) pair: latency (EWMA of recent p50/p95), error rate (rolling window), cost per token (from provider metadata), and capacity headroom (rate-limit budget remaining). Weights are themselves policy-configurable per routing strategy — "cheapest viable" weights cost heavily, "lowest latency" weights latency heavily, etc. Full spec in [docs/routing.md](docs/routing.md).

### How should observability work?
OpenTelemetry for traces and metrics is the baseline, wired at module boundaries (ingress, routing engine, adapter invocation) via interceptors so instrumentation isn't manually scattered through business logic. Langfuse receives LLM-specific trace data (prompt, completion, token counts, cost, model) for teams that want that view. See [docs/observability.md](docs/observability.md).

### How should authentication work?
API-key auth for simple adopters, OAuth2/OIDC for enterprise SSO integration, and mTLS for service-to-service traffic within a cluster, all pluggable via Spring Security. AuthZ is tenant/project scoped — every request resolves to a tenant, and routing policy, rate limits, and cost budgets are all tenant-scoped. See [docs/security.md](docs/security.md).

### How should future Kubernetes support work?
v1 ships standard Deployment/Service/HPA manifests and a Helm chart. A dedicated Kubernetes Operator (CRDs for `RoutingPolicy`, `ProviderConfig`) is a Phase 3 goal once the API surface for those resources has stabilized through the REST admin API first — we deliberately don't build a CRD-based API before we've validated the resource model via a simpler interface.

### How should AI-based routing integrate without modifying the router core?
By implementing `RoutingStrategy` (§4, `router-core.strategy`) exactly like any other strategy. An "AI-powered" strategy consumes the same inputs (policy, live provider scores, request features) as a rule-based strategy and returns the same ranked candidate list — the routing engine has no awareness that a model, rather than a heuristic, produced the ranking. This is precisely why the strategy interface exists: it lets us treat "smarter routing" as a swappable implementation detail, explored as a Stretch Goal, without ever requiring changes to `router-core.execution`, adapters, or the reliability layer.

## 8. Future Evolution

- Re-evaluate WebFlux vs. virtual-threads-on-MVC as Loom-based ecosystem tooling (structured concurrency, observability integration) matures.
- Revisit dynamic provider loading if/when a plugin-marketplace use case materializes.
- Consider a lightweight, dependency-minimal deployment mode (no Kafka/Postgres requirement) for smaller adopters, once core is stable.
- Evaluate a Go-based edge/sidecar variant for latency-critical, in-cluster local-inference-only routing, as a separate project rather than a core rewrite.

## 9. Open Discussion

This RFC is a starting point, not a final answer. Disagreement is expected and welcome — particularly on §4.1 (Java vs. Go) and §4.2 (WebFlux vs. MVC), which are the two decisions most likely to be revisited as the project and ecosystem mature.
