## 1. Summary

This RFC proposes the founding design of ModelRouter: an AI Inference Control Plane that decouples applications from inference providers via policy-driven routing. It documents *why* the project exists, the design philosophy behind it, the alternatives we considered and rejected, the tradeoffs we are consciously accepting, our assumptions, and how the design is expected to evolve. It also answers the critical design questions that must be resolved before Phase 1 implementation begins.

## 2. Why ModelRouter Exists

Teams integrating multiple LLM providers converge on the same ad-hoc solution: provider-selection `if/else` chains embedded in application code, duplicated per service, untested until a provider outage exposes the gap. This is the same problem HTTP microservices had before API gateways and service meshes existed — and the same category of solution applies: extract the cross-cutting concern (routing, retries, observability, policy) into shared infrastructure that applications call through, not around.

Nothing in the current OSS landscape treats this as a **control plane** problem. Most existing tools are either (a) a thin reverse proxy that normalizes request/response shapes across providers but does no policy-driven decision-making, or (b) an SDK-level abstraction that still runs *inside* the application process, so it doesn't solve duplication, centralized observability, or hot-reloadable policy. ModelRouter is deliberately infrastructure: a separately deployed, independently scaled service that owns the routing decision.

## 3. Design Philosophy

1. **Applications express intent, not destination.** A request says "complete this, under $X, within Yms, keep it on-prem if sensitive" — never "call OpenAI's gpt-4o." The provider is an implementation detail resolved at request time.
2. **The domain core knows nothing about any specific provider.** Ports & Adapters is non-negotiable; it's what makes "add a provider in an afternoon" true rather than aspirational.
3. **Reliability is structural, not incidental.** Retries, fallback, and circuit breaking are first-class concerns of the execution runtime, not something bolted on per-adapter.
4. **Observability is a design input, not an afterthought.** Every routing decision must be explainable after the fact: which candidates were considered, which was chosen, why, and what it cost.
5. **Boring where it counts.** We choose proven, operationally well-understood technology (Spring, Kafka, Redis, Postgres, Kubernetes) over novel infrastructure. The routing *policy* is where we innovate; the plumbing should not be a source of surprises.
6. **Extensibility over completeness.** v1 will not support every provider or every routing strategy. It must, however, make adding either trivial and safe.

## 4. Alternatives Considered

### 4.1 Language: Go vs. Java vs. Rust

#### Go

**Case for:** Dominant in cloud-native infrastructure (Kubernetes, Envoy, Istio). Lightweight concurrency (goroutines). Fast cold start. Small binary/memory footprint. Strong in the infrastructure-tooling ecosystem.

**Case against:**
- Go's type system makes DDD/Hexagonal patterns verbose — interfaces are satisfied implicitly, which aids decoupling but makes large dependency graphs harder to reason about at scale.
- The provider SDK ecosystem in Go is thinner than Java's. Most LLM providers ship Java (or Python) SDKs first; Go adapters would require more from-scratch HTTP client work.
- No equivalent to Spring's DI, lifecycle management, or declarative security — these would need to be built or adopted from smaller, less-proven libraries.
- The target adopter base (enterprise teams running Spring, Kafka, PostgreSQL, Kubernetes) already operates JVM infrastructure. A JVM control plane integrates into existing service meshes, secret managers, and CI/CD with less friction.

**Decision:** Rejected for v1 core.

**Tradeoff explicitly accepted:** JVM startup time (~2–4s vs. Go's ~50ms) and memory footprint (~200–400MB baseline vs. Go's ~20–50MB) are worse. This matters more for sidecar/edge deployment than for a centrally deployed control plane. If a sidecar form factor is needed (sub-second cold start, minimal footprint), a Go-based "ModelRouter Lite" is the right approach — as a separate project, not a rewrite of the core.

**Conditions for revisiting:** If the project pivots to a sidecar-first deployment model, or if the adopter base shifts away from JVM-centric infrastructure.

#### Rust

**Case for:** Zero-cost abstractions, memory safety without GC, excellent performance for proxy workloads (see Pingora).

**Case against:**
- Dramatically higher contributor barrier — Rust's learning curve would throttle OSS contributions.
- The provider SDK ecosystem in Rust is nearly nonexistent.
- Async Rust's ecosystem (tokio) is mature but still has rough edges for the kind of complex, multi-library integration ModelRouter requires (Spring Security equivalent, ORM, enterprise auth).
- Overkill for the workload — ModelRouter's hot path is I/O-bound (waiting on provider responses), not CPU-bound. The performance difference between Rust and Java 21 with virtual threads is negligible for this traffic shape.

**Decision:** Rejected. The contributor accessibility and ecosystem costs far outweigh the performance benefits for an I/O-bound control plane.

#### Decision: Java 21

**Rationale:**
- Java 21's virtual threads (Project Loom) close most of the concurrency-ergonomics gap with Go, while keeping a mature, batteries-included ecosystem (Spring, extensive provider SDKs, enterprise auth/observability integrations) that a small early-stage OSS project benefits from disproportionately.
- The DDD/Hexagonal architectural style this project commits to is more idiomatically supported by Java's type system and Spring's dependency injection than by Go's minimalism or Rust's trait system.
- Much of the target adopter base already operates JVM infrastructure.

### 4.2 Web Framework: WebFlux vs. Spring MVC vs. Vert.x vs. Quarkus Reactive

#### Spring MVC (thread-per-request)

**Case for:** Simpler programming model. Easier debugging (synchronous stack traces). Lower onboarding cost for contributors. Java 21 virtual threads make the thread-per-request model viable at higher concurrency than traditional platform threads.

**Case against:** Inference calls are long-lived (seconds to minutes for streaming), I/O-bound, and frequently hold a connection open for the request's full duration. Under thread-per-request, 1000+ concurrent in-flight requests require 1000+ threads. Virtual threads help with the thread *creation* cost but don't solve backpressure for streaming: MVC has no built-in mechanism for reactive backpressure on a token-by-token stream. Streaming ergonomics (SSE, chunked transfer) are bolted-on in MVC rather than first-class.

**Tradeoff if we'd chosen MVC:** Simpler code but fundamentally limited streaming support and no reactive backpressure. We'd need to build custom streaming infrastructure or accept degraded behavior under load.

#### Vert.x

**Case for:** Event-driven, non-blocking, polyglot. Lighter than Spring's full stack. Excellent raw performance in benchmarks.

**Case against:** Much smaller ecosystem. No equivalent to Spring Security, Spring Data, Spring Boot auto-configuration. Smaller contributor pool familiar with the framework. Would require building significant infrastructure (auth, config, lifecycle) that Spring provides out of the box.

**Decision:** Rejected. The ecosystem gap is too wide for a project that needs enterprise-grade auth, DI, and operational tooling from day one.

#### Quarkus Reactive (Mutiny)

**Case for:** Reactive + imperative hybrid. Fast startup (GraalVM native). Modern Java-first framework.

**Case against:** Mutiny's reactive API (`Uni`/`Multi`) is less established than Reactor's (`Mono`/`Flux`). Smaller community than Spring. GraalVM native compilation has sharp edges with reflection-heavy patterns (DI, proxies) that ModelRouter's architecture uses heavily. The ecosystem of provider SDKs tested against Quarkus is thinner.

**Decision:** Rejected. Not enough ecosystem maturity advantage over Spring WebFlux to justify the risk.

#### Decision: Spring Boot 3 + WebFlux (Reactor)

**Rationale:** WebFlux's non-blocking I/O lets a small number of event-loop threads service many concurrent long-lived connections with first-class backpressure, which is exactly the traffic shape ModelRouter serves. The reactive programming cost is concentrated in `router-ingress` and `core.execution` — the two modules where it's load-bearing — not spread across the whole codebase.

**Tradeoff explicitly accepted:** Reactive code is harder to write, debug, and onboard new contributors to than imperative MVC code. Provider adapters and admin/CQRS-side modules, which are not on the hot streaming path, may use simpler blocking patterns internally (wrapped via `Mono.fromCallable` + bounded elastic scheduler), trading a small amount of purity for contributor accessibility.

**Conditions for revisiting:** If Java's structured concurrency (still in preview) matures to provide equivalent streaming ergonomics and backpressure to Reactor, a migration to virtual-threads-on-MVC becomes viable. This is tracked as a future evolution item.

### 4.3 Routing Architecture: Monolithic Engine vs. Decomposed Subsystems

**Considered:** A single `RoutingEngine` class/module that handles policy resolution, candidate enumeration, scoring, strategy application, and execution in a single pipeline.

**Decision: Decomposed into three subsystems — Policy Engine, Execution Planner, Execution Runtime** (see ARCHITECTURE.md §3.2).

**Rationale:**
- **Testability:** Each subsystem has a distinct contract and can be tested in isolation. Policy merge semantics are tested without live providers. Planning (enumerate → score → rank) is tested without execution. Execution is tested with fake adapters and known candidate lists.
- **Failure isolation:** A bug in retry logic cannot corrupt candidate ranking. A bug in scoring cannot cause incorrect adapter invocation.
- **Extensibility boundary:** `RoutingStrategy` is a pure function that lives in the planner, taking scored candidates and returning an `ExecutionPlan`. If planning and execution were interleaved, strategy implementations would be tempted to inspect or influence execution state, breaking the pluggability contract.
- **Performance clarity:** The <30ms overhead budget can be decomposed and attributed: policy resolution (target: <2ms), planning (target: <15ms including Redis score fetch), execution orchestration overhead excluding provider call (target: <13ms).

**Tradeoff explicitly accepted:** Three subsystems mean three sets of interfaces, three modules, and more boilerplate than a single engine. For a v1 with 2-3 strategies and 2 providers, this is over-structured. We accept this because the decomposition pays for itself as soon as the system grows beyond the trivial case (Phase 2+), and retrofitting it later would be a painful refactor.

**Alternatives rejected:**
- *Two-part split (planning vs. execution only):* Considered, but policy resolution has distinct enough concerns (merge semantics, security constraint enforcement) that embedding it in either planning or execution would muddy that subsystem's contract.
- *Microservice split (separate processes):* Rejected — the latency of inter-process communication on the hot path would blow the <30ms budget. These are in-process module boundaries, not deployment boundaries.

### 4.4 Routing Execution Model: Synchronous vs. Asynchronous Decision-Making

**Considered:** Making the routing decision itself asynchronous/event-driven (e.g., publish a request event, have a separate routing service consume it and dispatch).

**Decision: Synchronous, in-process routing decision; asynchronous side effects only.**

**Rationale:** The routing decision (policy resolution → planning → candidate selection) must complete within the <30ms overhead budget and is fundamentally a request/response computation — introducing a message broker into the decision path would add latency (~5-20ms for Kafka round-trip) and failure modes (broker unavailability blocks all routing) for no benefit. What *is* asynchronous: usage metering, cache population, and score updates derived from the outcome, none of which the client needs to wait for.

**Tradeoff explicitly accepted:** Synchronous routing means a gateway pod's routing throughput is bounded by its ability to complete policy+planning within the budget. If policy resolution or score fetching becomes slow (e.g., Redis latency spike), routing is directly impacted. The mitigation is the L1 in-process cache (§4.6) which makes the common case a local memory read.

### 4.5 Provider Loading: Static vs. Dynamic

**Considered:** Java's `ServiceLoader`/OSGi-style dynamic plugin loading, allowing providers to be added/removed at runtime without a redeploy.

**Decision: Static compilation for v1** (each adapter is a Gradle module, wired via Spring's component scanning at boot), with adapter *configuration* (credentials, endpoints, enabled/disabled, routing weight) dynamic via `router-admin` + hot-reload.

**Rationale:** True dynamic class loading (hot-swapping adapter *code* at runtime) introduces:
- **Classloader isolation complexity** — ensuring adapter A's dependencies don't conflict with adapter B's requires OSGi-like module systems or custom classloaders, both of which are operationally fragile.
- **Security surface** — loading arbitrary code at runtime requires a trust/signing mechanism, sandboxing, and privilege separation that v1 doesn't need.
- **Testing complexity** — dynamic loading makes the set of adapters non-deterministic at test time, complicating CI.

New providers are added by developers who can ship a new gateway image via the normal release process. What genuinely needs to change at runtime is *configuration*, which we already support.

**Tradeoff explicitly accepted:** Adding a new provider requires a release and redeploy. For v1's expected cadence (2-3 new providers per Phase), this is acceptable. If a plugin marketplace or third-party adapter ecosystem emerges, dynamic loading becomes justified.

**Conditions for revisiting:** A concrete plugin-marketplace use case (§Stretch Goals in ROADMAP.md) with third-party adapter authors who cannot or should not contribute to the main repository.

### 4.6 Caching: Redis-only vs. Multi-tier

**Decision:** Redis as the shared L2 cache (prompt/response and provider-score cache), with a short-TTL (seconds) in-process L1 cache in each gateway pod.

**Alternatives considered:**
- **Redis-only (no L1):** Simpler, but adds a network round-trip (~0.5-1ms) to every routing decision for score lookup. At 1000+ req/s, this is ~1 second of aggregate network time per second, and creates a hard dependency on Redis latency for the hot path.
- **L1-only (no shared cache):** Each pod computes its own provider scores independently. Eliminates Redis dependency but means pods diverge in their view of provider health — a provider outage detected by pod A won't be reflected in pod B's routing until pod B also experiences the failure. Unacceptable for fast failover.
- **Memcached instead of Redis:** Memcached is simpler but lacks Redis's data structures (sorted sets for scoring, pub/sub for invalidation) and Lua scripting (for atomic rate-limit increments).

**Tradeoff explicitly accepted:** L1/L2 introduces eventual consistency — a pod may act on a provider score up to a few seconds stale. This is acceptable because provider scores are inherently a moving average of recent behavior, not a value requiring strong consistency. The staleness window (L1 TTL) is configurable and defaults to 3 seconds.

### 4.7 API Surface: REST vs. gRPC-only vs. REST+gRPC

**Alternatives considered:**
- **gRPC-only:** Best streaming performance (HTTP/2 multiplexing, binary framing, bidirectional streaming). But gRPC adoption outside backend services is limited — browser clients require gRPC-Web (a proxy layer), and many operations teams lack gRPC debugging tooling (vs. curl/Postman for REST).
- **REST-only:** Maximizes adoption ease and debuggability. But REST-based streaming (SSE) is unidirectional and has higher per-frame overhead than gRPC's binary framing, which matters for mobile SDKs on constrained networks.
- **GraphQL:** Considered for the admin API. Rejected — the admin API's resource model is simple CRUD, not the complex nested querying that justifies GraphQL's complexity.

**Decision:** REST (OpenAI-compatible-ish shape where sensible, for adoption ease) as the primary v1 surface, with gRPC as a fast-follow for internal/SDK-to-gateway traffic where streaming performance matters most (notably the Android SDK).

**Tradeoff explicitly accepted:** Maintaining two API surfaces (REST + gRPC) doubles the API contract surface area. Mitigated by a shared IDL-first generation step (Phase 2) that generates both from a single source of truth.

### 4.8 Observability: OTel-only vs. OTel + Langfuse

**Considered:** Using only OpenTelemetry for all observability, including LLM-specific traces.

**Decision: OTel for operational traces + metrics; Langfuse as a separate, opt-in export path for LLM-specific traces.**

**Rationale:** Operational traces (latency, error rates, routing decisions) should never contain prompt/completion content by default — this would make trace storage a compliance liability. Langfuse is purpose-built for LLM-specific tracing (prompt content, token usage, cost, eval) and provides tooling (prompt debugging, eval dashboards) that OTel backends don't. Keeping them separate means teams can enable OTel universally while restricting Langfuse access to teams that need prompt-level visibility, under appropriate data-handling policies.

**Tradeoff explicitly accepted:** Two export paths mean two sets of configuration, two potential failure modes, and two places to look when debugging. Mitigated by making Langfuse strictly opt-in (default: disabled) and by ensuring Langfuse failures never affect OTel or request serving (see [docs/failure-modes.md](../docs/failure-modes.md)).

## 5. Tradeoffs Accepted (Summary)

| Tradeoff | We accept it because | Conditions for revisiting |
|---|---|---|
| JVM footprint > Go footprint | Ecosystem fit and target-adopter alignment for centrally deployed control plane | Pivot to sidecar-first deployment model |
| Reactive complexity in hot-path modules | Backpressure and streaming correctness at 1000+ concurrent requests | Java structured concurrency reaches production maturity with streaming support |
| Three routing subsystems vs. one engine | Testability, failure isolation, and extensibility outweigh boilerplate cost | Never — this is a structural commitment |
| Eventual consistency on provider scores | Scores are a moving average; seconds-scale staleness is invisible to routing quality | Need for real-time score-based preemption (unlikely) |
| Static provider compilation (v1) | Avoids classloader/security complexity; new providers ship via normal release | Plugin marketplace use case materializes |
| No hosted multi-tenant SaaS in v1 | Focus on self-hosted core excellence first | Core is proven stable, demand for hosted offering emerges |
| Dual API surface (REST + gRPC) | REST for adoption, gRPC for performance; neither alone suffices | Industry converges on a single protocol for this traffic shape |
| Dual observability (OTel + Langfuse) | Compliance separation of operational vs. prompt-content traces | OTel ecosystem adds first-class LLM trace support |

## 6. Assumptions

- Adopters run Kubernetes (or are willing to) — the deployment model is not optimized for bare-VM or serverless-first environments in v1.
- Adopters have (or are willing to run) Redis, PostgreSQL, and Kafka — no attempt is made to support a "zero external dependencies" mode in v1; an embedded/lite mode is a plausible future variant, not a v1 goal.
- The majority of early traffic is text completion/chat-style inference, not multi-modal; multi-modal routing (image/audio-aware provider capability negotiation) is designed for extensibility but not optimized in v1.
- Consumers of the API are backend services and mobile apps (hence Java + Android SDKs first), not primarily browser-based frontends calling ModelRouter directly.
- Provider APIs are externally stable enough that adapter maintenance is bounded — a provider breaking their API in an incompatible way is a provider bug, not a ModelRouter design failure.

## 7. Critical Design Questions

### Why WebFlux over MVC?
Answered in §4.2. Streaming + high concurrency + backpressure requirements make non-blocking I/O the correct default for the hot path; blocking code remains acceptable in non-hot-path modules.

### Why Java instead of Go or Rust?
Answered in §4.1. Ecosystem fit, virtual threads closing the concurrency gap, target-adopter infrastructure alignment, and contributor accessibility.

### Why three routing subsystems instead of one engine?
Answered in §4.3. Testability, failure isolation, and extensibility boundary. The <30ms budget is decomposed across the three subsystems.

### Should routing be synchronous?
Yes, for the decision itself (§4.4). Side effects (metering, cache population, score updates) are asynchronous via Kafka.

### How should streaming work?
`router-ingress` exposes SSE (REST) and server-streaming gRPC. Internally, `InferenceProvider.invokeStreaming()` returns a `Flux<InferenceChunk>` that flows through cache (write-through, populated once the stream completes), retry/fallback logic (which can only apply *before* the first chunk is emitted — once bytes have reached the client, a fallback would require client-side replay semantics, which is out of scope for v1 and explicitly documented as a limitation in [docs/failure-modes.md](../docs/failure-modes.md)), and normalization, before being re-framed to the client's chosen protocol.

### Should routing strategies be pluggable?
Yes — `RoutingStrategy` is a port in `core.planner` (ARCHITECTURE.md §3.2.2). Shipping strategies (lowest-latency, cheapest-viable, weighted round robin, privacy-constrained) are first-party implementations of the same interface a third party could implement.

### Should providers be loaded dynamically?
No, not at the code level, for v1 — see §4.5. Configuration is dynamic; code is statically compiled per release.

### Should provider metadata be cached?
Yes. Static metadata (context window, supported modalities, pricing table) is cached with a longer TTL (minutes) and refreshed via `router-admin`; live metadata (current latency, error rate, health) is cached with a short TTL (seconds) as described in §4.6.

### How should retries work?
Retry policy is part of `RoutingPolicy`, resolved per request by the policy engine. The execution runtime enforces: max attempts, backoff strategy, which failure classes are retryable (429/503/timeout are retryable, 400/401 are not). Retries are scoped to the *pre-first-byte* window (see streaming answer). Retries consume the `ExecutionPlan` produced by the execution planner, advancing to the next candidate on exhaustion of a given provider's retry budget — this is what makes retry and fallback the same mechanism rather than two.

### How should health checks work?
Two tiers: **passive** (every real request's outcome updates the provider's live score/circuit-breaker state — the primary signal) and **active** (a low-frequency background prober, per provider, to detect recovery from an open circuit before real traffic would). Health state lives in Redis, shared across gateway pods.

### How should provider scoring work?
A composite, weighted score per (provider, model) pair: latency (EWMA of recent p50/p95), error rate (rolling window), cost per token (from provider metadata), and capacity headroom (rate-limit budget remaining). Weights are themselves policy-configurable per routing strategy — "cheapest viable" weights cost heavily, "lowest latency" weights latency heavily, etc. Full spec in [docs/routing.md](../docs/routing.md).

### How should observability work?
See §4.8 and [docs/observability.md](../docs/observability.md).

### How should authentication work?
API-key auth for simple adopters, OAuth2/OIDC for enterprise SSO integration, and mTLS for service-to-service traffic within a cluster, all pluggable via Spring Security. AuthZ is tenant/project scoped. See [docs/security.md](../docs/security.md).

### How should future Kubernetes support work?
v1 ships standard Deployment/Service/HPA manifests and a Helm chart. A dedicated Kubernetes Operator (CRDs for `RoutingPolicy`, `ProviderConfig`) is a Phase 3 goal once the API surface for those resources has stabilized through the REST admin API first — we deliberately don't build a CRD-based API before we've validated the resource model via a simpler interface.

### How should AI-based routing integrate without modifying the router core?
By implementing `RoutingStrategy` (ARCHITECTURE.md §3.2.2) exactly like any other strategy. An `AdaptiveRoutingEngine` strategy consumes the same inputs (policy, live provider scores, request features) as a rule-based strategy and returns the same `ExecutionPlan`. This adaptive routing strategy will be informed by `FeedbackEvent`s (e.g., user satisfaction, task success metrics) provided asynchronously by the client application. The execution planner has no awareness that a model, rather than a heuristic, produced the ranking. This is precisely why the strategy interface exists.

### How should request analysis work?
Request Analysis (e.g. `RequestAnalyzer`) should be a pluggable abstraction inside the Execution Planner. We explicitly avoid hardcoding models like a Random Forest gateway here. The goal is to provide an extension point where adopters can plug in rules, ML classifiers, or LLM-based classifiers to categorize semantic intent, complexity, or safety, which then informs the Routing Policy.

### What about context optimization?
A `ContextOptimizer` is proposed as a future RFC. It will act as a policy-driven pre-execution step that applies transformations to the prompt context (summarization, compression, trimming) before routing, optionally influenced by external `FeedbackEvent`s. It is completely independent of the routing logic itself.

## 8. Future Evolution

- Re-evaluate WebFlux vs. virtual-threads-on-MVC as Loom-based ecosystem tooling (structured concurrency, observability integration) matures.
- Revisit dynamic provider loading if/when a plugin-marketplace use case materializes.
- Consider a lightweight, dependency-minimal deployment mode (no Kafka/Postgres requirement) for smaller adopters, once core is stable.
- Evaluate a Go-based edge/sidecar variant for latency-critical, in-cluster local-inference-only routing, as a separate project rather than a core rewrite.
- Investigate WebTransport as a potential replacement for SSE+gRPC dual streaming once browser and tooling support matures.

## 9. Open Discussion

This RFC is a starting point, not a final answer. Disagreement is expected and welcome — particularly on:

- §4.1 (Java vs. Go) — the decision most likely to be revisited as the project and ecosystem mature.
- §4.2 (WebFlux vs. MVC) — virtual threads may make this moot within 1-2 years.
- §4.3 (Three subsystems) — is the decomposition premature for v1's scope, or is it structural insurance against v2 pain?
- §4.7 (REST + gRPC) — is maintaining two API surfaces worth the operational cost?

File an issue with the `design-review` label or use the [design review issue template](../.github/ISSUE_TEMPLATE/design_review.md).
